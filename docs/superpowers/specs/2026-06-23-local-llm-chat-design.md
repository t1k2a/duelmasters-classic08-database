# ローカルLLM対話機能（デュエマクラシック AIアシスタント）設計

- 作成日: 2026-06-23
- ステータス: ドラフト（ユーザーレビュー前 / 技術裏取り反映済み 2026-06-23）
- 関連: 既存の静的サイト（GitHub Pages）/ `public/cards.json` / `src/api/server.ts`

> 本ドキュメントはブレストで合意した内容の清書。技術前提は researcher の裏取り済み（§12参照）。

## 1. 目的とスコープ

公開サイト（GitHub Pages）にデュエマクラシックに関する**対話型アシスタント**を追加する。回答はローカルLLM（Ollama）で生成し、**提供者（ユーザー）のマシンが起動している間だけ**利用可能とする。

### 主目的（優先度順）
1. **デッキ構築相談（メイン）** — アーキタイプ/レシピ/メタを踏まえた助言。
2. **カード事実Q&A** — 収録カードの能力・コスト・文明・種族・殿堂を正確に答える。
3. **ルール・用語・環境解説** — ブロッカー/殿堂/05と08の違い等。
4. **雑談・その他** — 軽く対応。

### 最重要要件
- **情報の相違ゼロ**: 誤ったカード事実を出さない。LLM単体では保証不能なため「**事実をLLMに生成させない**」設計で担保する（§3）。
- **厳格モード**: DBで裏が取れないことは「このDBには情報がありません」と答え、推測しない。

### 非スコープ（YAGNI）
- 合言葉/認証ゲート（将来オプション）。
- RAG第2段の意味検索（埋め込み）は初版では任意・後追加。
- 取得不可なDMC専用カードの拡充（別件）。

## 2. アーキテクチャ

```
[公開サイト GitHub Pages]            [自宅マシン（提供者）]
  public/index.html                   ┌──────────────────────────┐
   - 💬 AIに聞く パネル                │ [3] チャットサーバ(新規)   │
   - health で生死判定        fetch   │  src/chat/server.ts       │
   - ストリーミング描画  ───ngrok───▶ │  Hono / Postgres非依存     │
                                       │  public/*.json を直接ロード│
                                       │  RAG → 厳格プロンプト構築   │
                                       │  Ollamaへ中継・ストリーム   │
                                       └─────────┬────────────────┘
                                                 │ localhost:11434
                                       ┌─────────▼────────────────┐
                                       │ [4] Ollama (qwen2.5:7b)   │
                                       └──────────────────────────┘
```

### 構成要素
1. **フロント**（`public/index.html` に追加）: チャットUI、health生死判定、ストリーミング表示、カードカード/根拠リンク描画。
2. **つなぎ**: ngrok（**無料の固定ドメイン1個**）。HTTPS終端されGitHub Pages(HTTPS)からのfetchで mixed-content は発生しない。**無料枠は警告ページ(interstitial)が入るため、フロントのfetchに `ngrok-skip-browser-warning` ヘッダ必須**。CORSはngrokが付与しないので**Hono側で明示**。無料制限: 帯域1GB/月・20kリクエスト/月（ストリームのバイト量に注意）。
3. **チャットサーバ**（新規 `src/chat/server.ts`、Hono、**Postgres非依存**）: `public/cards.json`・`recipes.json`・`meta-decks.json`・`hall-of-fame.json`・`knowledge/*.md` をメモリにロード。`/api/chat`・`/api/health`。同時実行直列化＋レート制限。CORSヘッダを明示付与。
4. **推論**: Ollama（**`qwen2.5:7b`（Q4_K_M, 約4.7GB, 8GBに収まり日本語◎）を第一候補確定**。`gemma2:9b` は8GBではVRAM厳しめ＝部分CPUオフロードで低速になりやすい。軽量保険に `qwen2.5:3b`。差し替え可）。`localhost:11434` の `/api/chat`（NDJSONストリーム）を使用。

### 重要判断: なぜ既存 `src/api/server.ts` を使わないか
既存サーバは Prisma/PostgreSQL 依存で、自宅マシンで常時 Postgres を動かすのは非現実的（Docker起動できない環境実績あり）。チャットサーバはビルド成果物 `public/*.json` を直接読む**DB非依存の別サービス**にする。

## 3. 「情報相違ゼロ」の担保（3本柱）

1. **カード事実はLLMに書かせない**: RAGで抽出したカードの実データを構造化して返し、フロントが**カードカードとして実値描画**。LLMは《カード名》参照のみで、コスト/パワー/能力テキスト等の数値・文言を生成しない。
2. **文脈外は答えない（厳格システムプロンプト）**: 「渡した文脈のみを根拠に。無ければ『このDBには情報がありません』。カードの数値や能力を創作しない。外部知識を使わない。」**温度 0〜0.2**（決定論寄り）。カード名・数値を返す箇所は必要に応じ Ollama の `format:json`（structured output）で縛る。
3. **根拠を必ず提示**: 使ったカード/レシピ/知識をUIに列挙し、既存のカード詳細・レシピページへリンク。

## 4. データ（grounding corpus）

| ソース | 用途 | 区分 |
|---|---|---|
| `public/cards.json`（2,171枚） | カード事実 | 既存 |
| `public/data/recipes.json`（462件） | デッキ構築の実例 | 既存 |
| `public/data/meta-decks.json` | アーキタイプ解説 | 既存 |
| `public/data/hall-of-fame.json` | 殿堂レギュ | 既存 |
| `knowledge/*.md`（新規・少量手書き） | ルール・用語・環境解説 | **新規** |

- `knowledge/` が無いと厳格モードではルール質問(C)を断り続けるため、用語10〜20件程度から整備する（例: ブロッカー、シールド・トリガー、殿堂/プレミアム殿堂、進化、クロスギア、ゴッド・リンク、クラシック05と08の違い 等）。

### 検索（RAG）の方針
- **第1段（語彙ベース・必須）**: カード名あいまい一致（既存のかな正規化検索を流用）＋フィルタ解釈（文明/コスト/種族等）→ 該当カードを**完全一致で抽出**。"相違ゼロ" の主役。
- **第2段（意味ベース・任意/後追加）**: レシピ説明や `knowledge/` の意味検索に Ollama 埋め込みモデル（`nomic-embed-text` 等）。初版は語彙のみで開始。

## 5. データの流れ

### 例①: カード事実「ボルメテウス・ホワイト・ドラゴンの能力は？」
1. カード名検出（かな正規化あいまい検索）→ `cards.json` から該当を完全一致抽出
2. プロンプト = 厳格指示 ＋ カードデータ ＋ 質問
3. Ollama: 地の文のみ生成（数値は出さない）
4. 返却 = 地の文 ＋ カード構造データ → フロントが能力等をDB実値で描画

### 例②: デッキ構築（メイン）「クラシック08で天門デッキを組みたい。何を軸に？」
1. 意図判定: デッキ構築 ＋ アーキタイプ「天門」
2. 多段検索: `meta-decks.json`（天門の解説/キーカード/戦略）＋ `recipes.json`（天門系の実戦レシピ2〜3件）＋ `cards.json`（キーカード実データ抽出）
3. プロンプト構築: 厳格指示 ＋（メタ解説＋レシピ骨子＋キーカード事実）＋ 質問
4. Ollama 生成: 渡したカード/レシピの範囲だけで助言
5. 構造化返却: キーカードをカードカードで、参照レシピをリンクで「根拠」として同梱
6. フロント描画: 助言の地の文 ＋ キーカード（実値）＋ 根拠レシピリンク
7. DB外要求時: 「このDBには情報がありません」＋近い収録カード提案にとどめる

## 6. UX

- **入口**: デッキFAB付近に「💬 AIに聞く」ボタン → ボトムシート（既存 `#deckPanel`/`#iosInstallSheet` パターン）。
- **生死判定**: 起動時 `health` → 生きていれば通常、落ちていれば入口グレーアウト＋「今は使えません（提供者PCオフライン）」。
- **チャット**: 吹き出し（ユーザー/アシスタント）、応答は**トークン単位ストリーミング**。応答下にカードカード（実値）＋根拠リンク。入力例チップ（「天門デッキを組みたい」等）、会話リセット、モバイル優先。
- **ストリーミング受信は `fetch` + `ReadableStream` を使う（EventSource は不可）**。理由: ngrok無料枠の警告回避に `ngrok-skip-browser-warning` ヘッダが要るが、EventSource はカスタムヘッダを付けられないため。`health` fetch にも同ヘッダを付与。
- **会話履歴**: **軽いマルチターン**＝直近2〜3往復のみ文脈に含める（速度維持）。

## 7. 公開運用の安全面

- **同時実行**: Ollama生成は**同時1件に直列化**（single-flight キュー）。待ち人数表示、深すぎる待ち行列は却下。生成タイムアウト60秒。
- **レート制限**: IP単位（例 10回/分・100回/時、小規模なので緩め）。入力長上限（例 500文字）。
- **ngrok**: チャットサーバを `localhost:8788` 等に bind → ngrok 固定ドメイン（`ngrok http --url=<固定>.ngrok-free.app 8788`）で公開。フロントは接続先を1定数（`CHAT_API_BASE`）で保持。**全リクエストに `ngrok-skip-browser-warning` ヘッダを付与**（無料枠の警告ページ回避）。無料枠制限（1GB・20kリクエスト/月）に収まるよう、ストリームのバイト量と無駄打ちに留意。
- **CORS**: Hono側で `Access-Control-Allow-Origin: https://t1k2a.github.io`（＋dev用 localhost）を明示。プリフライト(OPTIONS)対応。ngrokはCORSを付けない。
- **注意**: ngrok URL は公開JSに載るため誰でも到達可。守りは CORS＋レート制限＋同時実行キャップ。さらに絞るなら合言葉ゲートを後付け（初版見送り）。
- **プライバシー**: 個人情報を残さない（件数程度の集計のみ）。
- **プロンプトインジェクション**: カード事実は決定論描画のため誤情報は出ない。システムプロンプトは強めに固定。

## 8. エラー処理

| 状況 | サーバ | フロント |
|---|---|---|
| Ollama 落ち/未起動 | health失敗 / chat 503 | 「今は使えません（提供者PCオフライン）」 |
| モデル未取得 | エラー検出＋ログに `ollama pull qwen2.5:7b` 案内 | 「今は使えません」 |
| 生成タイムアウト(60s) | 打ち切り | 「時間がかかっています。もう一度」 |
| 順番待ち過多/レート超過 | 429 | 「混んでいます/少し待って」 |
| ngrok/回線断 | fetch失敗 | オフラインUIと同じ |
| DB外質問（厳格） | 200 | 「このDBには情報がありません」＋近い収録カード提案 |
| 過大/空入力 | 400 | 入力案内 |

## 9. テスト戦略

LLM出力は非決定的なので、**決定論的な部分**を検証する。
- **ユニット**: RAG抽出（質問→期待カード/レシピ）、カード名あいまい一致、フィルタ解釈、厳格リフューザル判定。
- **コントラクト**: `/api/health`・`/api/chat` を **Ollamaスタブ**で検証（ストリーム形・根拠・カード構造データ・リフューズ経路）。CIで実LLM非依存。
- **フロント(Playwright)**: パネル開閉、オフライン状態（health downモック）、ストリーミング描画、カードカード描画、各エラー表示。
- **精度スモーク（“相違ゼロ”の要）**: ゴールデンQ&A少数を実Ollamaでローカル実行。カード事実がDBと完全一致するか・デッキ質問が実在レシピを引くかを目視＋自動照合。

## 10. 構成要素ごとの責務（独立性）

| ユニット | 責務 | 入力→出力 | 依存 |
|---|---|---|---|
| `corpus-loader` | `public/*.json`＋`knowledge/*.md` をロード・正規化 | 起動時→メモリ索引 | fs |
| `retriever`（RAG） | 質問→関連カード/レシピ/知識を抽出 | 質問文→根拠セット | corpus, 既存検索ロジック |
| `prompt-builder` | 厳格システム＋文脈＋履歴を組む | 根拠＋履歴＋質問→プロンプト | — |
| `ollama-client` | Ollama `/api/chat`(stream:true) を fetch、NDJSONを逐次パース | プロンプト→トークン列 | Ollama |
| `chat-route` | キュー/レート制限/タイムアウト/CORS、NDJSON→SSE(or chunked)中継、`ngrok-skip`許容 | HTTP→SSE/chunked | 上記すべて |
| `health-route` | 生死＋モデル名＋待ち人数 | HTTP→JSON | ollama-client |
| フロント `chat-ui` | 入口/生死/ストリーミング/カードカード/根拠描画 | ユーザー操作→表示 | chat-route |

## 11. 起動手順（提供者）

1. `ollama serve`（＋初回 `ollama pull qwen2.5:7b`）
2. `npm run chat`（新チャットサーバ）
3. `ngrok http 8788`（固定ドメイン指定）

READMEに上記＋「マシンを落とすとチャットは自動的に利用不可になる」旨を記載。

## 12. 技術前提（researcher裏取り済み 2026-06-23）

**確定事項**
- **ngrok 無料枠**: 固定ドメイン1個可（`ngrok http --url=<固定>.ngrok-free.app 8788`）。HTTPS終端されmixed-content無し。制限=帯域1GB/月・20kリクエスト/月・同時endpoint3個。
  - ⚠️ 無料枠は警告ページ(interstitial)が入る → fetchに **`ngrok-skip-browser-warning` ヘッダ必須**。
  - ⚠️ **EventSourceはカスタムヘッダ不可** → ブラウザは **fetch+ReadableStream** で受信。
  - ⚠️ CORSはngrokが付けない → **Hono側で明示**。
- **モデル**: `qwen2.5:7b`(Q4_K_M, 約4.7GB, 32K context) が8GBに収まり日本語◎で**第一候補**。`gemma2:9b` は8GBでVRAM厳しめ（部分CPUオフロードで低速化）。軽量保険 `qwen2.5:3b`。tokens/sec実数は要実測（推測: フルGPUで数十tok/s、CPUオフロード時は一桁台）。
- **API/中継**: `/api/chat`(messages配列, stream:true, NDJSON) を使用。Hono が fetch受け→ `streamSSE`/chunked でブラウザへ中継。OpenAI互換 `/v1` は experimental なので native `/api/chat` を採用。
- **埋め込み（第2段RAG）**: 日本語意味検索なら `bge-m3`。ただしチャットLLMとのVRAM同時常駐はモデル切替ロードでレイテンシ増のため運用要検証。**初版は第1段（語彙/構造化）のみで開始し後追加**。
- **グラウンディング**: 温度0〜0.2＋強いシステムプロンプト（context外禁止/数値創作禁止/無ければ不明）＋必要に応じ `format:json`。小型LLMほど低温と縛りが効く。

**設計フェーズで詰める残論点（writing-plansで決定）**
- ブラウザ⇄Hono間のストリーム形式: Hono `streamSSE` か 生 chunked か（どちらも fetch+ReadableStream で受信可）。
- 埋め込み(第2段RAG)を初版に入れるか、語彙のみで出すか。
- `format:json` 構造化をどの応答に適用するか（カード抽出のみ等）。
