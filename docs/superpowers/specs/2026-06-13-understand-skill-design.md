# Skill「understand」設計ドキュメント

- 日付: 2026-06-13
- ステータス: 設計確定（実装計画待ち）
- 成果物: 理解しながら実装／読解するための Skill（SKILL.md）。完成後 GitHub に push。

## 背景と課題

実装を Claude にほぼ任せた結果、コードは動くがユーザー自身が技術仕様・設計の意図を把握できていない。「実装を進めながら、同時にユーザーの理解も深まる」状態を Skill として仕組み化したい。

## ゴール（このSkillが防ぎたい一番の失敗）

**後で自分でコードを保守・改修できない**こと。
→ 学術的な設計理論ではなく、**保守者の視点**での理解（読んで動かせる・どこを変えればどう動くか分かる）を得ることが目的。

## 確定した方針（ブレスト結果）

| 論点 | 決定 |
|---|---|
| 主目的 | 保守・改修できる理解（読んで動かせる） |
| 発動 | **手動トグル**（`/understand` で起動。普段は通常フロー） |
| タイミング | **両対応**（これから作る=pre / もうある=post） |
| 成果物ファイル | **作らない**（対話の中で理解を完結。軽量） |
| ペルソナ | **メンター寄り**（伴走する先輩エンジニア。解説＋確認テスト） |

## 中核思想

「説明した＝理解した」を信じない。各節目で**逆質問して検証**する。検証は常にメンテナの問いの形を取る：

> 「後で〇〇を変えるなら、どこを触る？／この値が来なかったら何が起きる？」

## 動作フロー

### 1. 入口判定
起動時に「これから作る(pre)／もうある(post)」を判定する。
- **pre**: 小さなチャンクに分けて実装しながら進む
- **post**: 対象コードを読み、保守者目線でチャンクごとに読み解く

### 2. チャンク分割
1ファイル／1関数／1データフロー単位に割る。1チャンク = 1チェックポイント。
（大きすぎる単位を一度に扱わない。）

### 3. チェックポイント儀式（中核）
各チャンクで以下を順に行う：

| 段階 | Claude がやること |
|---|---|
| ① 何を・どこで | このチャンクの役割と場所（`file:line`）、データの入出力を簡潔に |
| ② なぜ・代替 | なぜこの方法か（自明な代替案より良い理由）を1〜2行 |
| ③ 逆質問 | メンテナの問いを投げ、**ユーザーに答えさせる** |
| ④ 判定 | 答えられたら次へ。詰まったら補足→軽く再確認→次へ |

逆質問テンプレ（③）の例：
- 「後で○○の挙動を変えたくなったら、どのファイルのどこを触る？」
- 「この関数に△△が渡されなかったら何が起きる？」
- 「このデータはどこから来て、どこへ流れていく？」

### 4. 軽量化の逃げ道
ユーザーが「浅くて／飛ばして」と言えば③をスキップ。**些末な変更にまで重い検証をしない**。
（= Skill が嫌われて無効化されるのを防ぐ最重要装置。）

### 5. 締め
最後に統合質問を1つ：「この機能を改修するなら入口はどこ？」でメンタルマップの成立を確認する。**ファイルは作らない。**

## 既存 `@dev` pair/solo モードとの関係

直交させる。役割が違う：
- pair/solo = 「**誰が手を動かすか**」の制御
- `/understand` = 「**理解が残るか**」の制御

pair モード中なら従来どおり Claude は実装ファイルを書かず提案に留め、その上でチェックポイントを重ねる。`/understand` は pair/solo を上書きしない。

## SKILL.md に盛り込むアンチパターン（明示して効かせる）

- ❌ 説明して終わり（検証しない）
- ❌ チャンクが大きすぎる（一度に全部）
- ❌ ファイルを作る
- ❌ 些末な変更にまで重い検証を強制する
