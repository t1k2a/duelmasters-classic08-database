👤 ユーザーストーリー

As 08環境プレイヤー

I want 詳細画面からそのカードを実際に使う合法レシピを推薦され、気に入ればデッキビルダーに読み込み、自分のレシピも投稿したい

So that 機械生成でなく実プレイヤーの感覚で組まれた信頼できるデッキを出発点にでき、共有できる

✅ 受け入れ条件 (AC-1 〜 AC-8)

AC-1

エントリーポイント

詳細モーダルに「📋 このカードを使うデッキを見る」ボタン＋マッチ件数を表示する。

AC-2

推薦リスト表示・読込

  • レシピカード: デッキ名 / 文明バッジ / アーキタイプ / 採用枚数 / 出典URL
  • 「デッキビルダーに読込」→ 既存 deck 配列に流込 → 上書き確認ダイアログ
  • 出典 URL の表示は必須（Wayback リンク）
  • 表示順: 初期=採用枚数降順
AC-3

合法性保証

validated: true のレシピのみ表示。全カードが cards.json に存在すること（= Classic08準拠フィルタ）。

AC-4

0件フォールバック

  1. シードカード直接マッチ → 0件なら同種族/同文明のレシピを提示
  2. それも0件なら「レシピがありません → 投稿する」導線を表示
AC-5 ★

HOF_MAP ロードタイミング保証（最重要）

既存 index.html:296 の fetch は .catch() で握りつぶし。失敗時 HOF_MAP が空のまま残り、殿堂チェックがすり抜ける危険がある。

  • AC-5-1: ロード完了まで「読込」ボタンを disabled
  • AC-5-2: 押下時に未ロードなら完了を待ってから実行
  • AC-5-3: generateDeck()/loadRecipe() 冒頭で HOF_MAP.size === 0 なら中断＋「読込中」表示
  • 判定は name 照合のみ・id 照合禁止（HOF の id は null 混在）
AC-6

ユーザー投稿 — GitHub PR プレフィル

サーバー不要。フォーム入力 → JSON 断片生成 → GitHub PR/Issue 作成 URL にプレフィルして新規タブで開く。オーナーがレビュー・マージ。

AC-7

投稿時クライアントバリデーション

コードチェック内容
V1合計ちょうど40枚
V2同名カード ≤ 4枚
V3殿堂カード ≤ 1枚 (name照合)
V4プレミアム殿堂 = 0枚
V5全カードが cards.json (08環境) に存在

不合格は違反内容を明示してブロック。合格後に PR URL を生成。

AC-8

パフォーマンス

最大8,000件の recipes.json 読込・フィルタが UI をブロックしない。gzip 圧縮サイズを確認し、必要なら遅延ロード・分割を検討。
